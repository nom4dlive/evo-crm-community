## 2. High Severity Finding: Authentication Service (evo-auth) Tenant Leakage

### Finding Description
While logged in as a tenant administrator (`teste@bodyharmony.tech` under tenant `bodyharmony-test`), the user can view, edit, and delete users from the master tenant (`bodyharmony`) on the `https://crm.bodyharmony.tech/settings/users` page. 

### Root Cause Analysis
1. **Unscoped Controller Query**: In `evo-auth-service-community/app/controllers/api/v1/users_controller.rb`, the `users` helper method queries all users globally:
   ```ruby
   def users
     @users ||= User.order_by_full_name.includes(:user_roles)
   end
   ```
   It has no scoping restriction (e.g. `where(account_id: current_user.account_id)`).
2. **Missing `AccountScoped` Concern**: Unlike the CRM backend (`evo-crm`), which utilizes a shared `AccountScoped` concern to automatically apply tenant scopes to ActiveRecord queries, the authentication service (`evo-auth`) does not scope `User` or `UserRole` queries to the active tenant.
3. **AgentBuilder Account Omission**: Newly invited users created through `AgentBuilder` are not associated with any `account_id` because `user.account_id = inviter.account_id` is never set during instantiation.

### Recommended Solution
1. **Add Tenant Scoping to UsersController**:
   ```ruby
   def users
     @users ||= if current_user.account_id.present?
                  User.where(account_id: current_user.account_id)
                else
                  User.all
                end.order_by_full_name.includes(:user_roles)
   end
   ```
2. **Fix Account Association in AgentBuilder**:
   ```ruby
   user = User.new(email: email, name: name, password: generated_password)
   user.account_id = inviter.account_id if inviter&.account_id.present?
   ```
